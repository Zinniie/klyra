## Project Overview

Klyra is a Flutter fintech wallet app targeting iOS, Android, Web, Linux, macOS, and Windows. It uses Firebase (Auth + Firestore + FCM) for backend services and Stripe for payments.

## Commands

```bash
# Install dependencies
flutter pub get

# Run the app (pass Stripe key via dart-define)
flutter run --dart-define=STRIPE_PUBLISHABLE_KEY=pk_test_...

# Lint / static analysis
flutter analyze

# Run all tests
flutter test

# Run a single test file
flutter test test/features/auth/bloc/auth_bloc_test.dart

# Run tests with coverage
flutter test --coverage

# Code generation (freezed, injectable, json_serializable)
dart run build_runner build --delete-conflicting-outputs

# Firebase config regeneration
flutterfire configure

# Release builds
flutter build apk --release --dart-define=STRIPE_PUBLISHABLE_KEY=pk_live_...
flutter build ios --release --dart-define=STRIPE_PUBLISHABLE_KEY=pk_live_...
```

## Architecture

### Feature Structure

Each feature under `lib/features/` follows Clean Architecture with three layers:

```
features/<name>/
  screens/   # Flutter widgets (UI only, no business logic)
  bloc/      # BLoC: events, states, bloc class
  repository/ # Data access: Firebase, Stripe, local storage
```

### State Management (BLoC)

All state lives in BLoCs. The pattern is:
1. Widget dispatches an **Event** → `context.read<XBloc>().add(XEvent())`
2. BLoC handler calls **Repository** method
3. Repository returns data from Firebase / Stripe / secure storage
4. BLoC emits new **State**
5. Widgets rebuild via `BlocBuilder` / `BlocListener`

`AuthBloc` (`lib/features/auth/bloc/auth_bloc.dart`) is the most complete example to follow.

### Providers

BLoCs and repositories are wired in `main.dart` via `MultiRepositoryProvider` wrapping `MultiBlocProvider`. Add new repositories to the repository provider list before adding their dependent BLoCs.

### Navigation

`go_router` (`lib/core/router/router.dart`) handles all routing with auth-based redirect guards. Protected routes check `AuthBloc` state; unauthenticated users are redirected to the auth flow.

### Data Models

Domain models live in `lib/core/models/models.dart`:
- `KlyraUser` — uid, balance, currency, KYC status, biometric flag
- `KlyraTransaction` — type (send/receive/topUp), status, amounts, timestamps
- `KlyraCard` — wraps Stripe payment method (brand, last4, expiry)

### Firebase Structure

Firestore collections: `users`, `transactions`, `cards`, `notifications`.

### Design System

Theme is defined in `lib/core/theme/app_theme.dart`:
- **Fonts**: Fraunces (display/headers) + DM Sans (body)
- **Spacing constants**: `xs`=4, `sm`=8, `md`=16, `lg`=24, `xl`=32, `xxl`=48, `xxxl`=64
- **Key colors**: Navy `#0D1B2A` (primary), Teal `#1D9E75` (accent)
- Use `AppTheme.amountLarge/Medium/Small` text styles for any currency display

## Implementation Status

Many features are scaffolded but not yet implemented. Before adding new code, check whether the repository class and BLoC for that feature exist — many are empty stubs.

**Implemented**: `AuthBloc`, `HomeScreen`, `BalanceCard`, `SendMoneyScreen` (4-step wizard)

**Missing repositories** (need Firebase/Stripe integration): `AuthRepository`, `TransactionRepository`, `CardRepository`

**Missing BLoCs**: `TransactionBloc`, `CardBloc`, `ProfileBloc`

**Missing screens**: auth flow screens, `TopUpScreen`, `TransactionHistoryScreen`, `TransactionDetailScreen`, `CardsScreen`, `AddCardScreen`, `ProfileScreen`

## Environment

- Stripe publishable key is passed via `--dart-define=STRIPE_PUBLISHABLE_KEY=...` (never hardcoded)
- Firebase config is generated by `flutterfire configure` into `lib/firebase_options.dart` (git-ignored)
- Sensitive files excluded from git: `.env`, `firebase_options.dart`, `google-services.json`, `GoogleService-Info.plist`
